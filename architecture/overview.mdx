## Introduction

OpsChat is built on a modern, scalable architecture designed for real-time communication. The system follows a client-server model with WebSocket support for bidirectional communication, enabling instant message delivery and presence updates.

<Note>
The architecture is designed to support horizontal scaling with multiple backend instances using Redis as a pub/sub adapter.
</Note>

## Request Flow

### REST API Request

<Steps>
  <Step title="Client sends HTTP request">
    Frontend makes API call to `/api/*` endpoint
  </Step>
  
  <Step title="Ingress routes to backend">
    NGINX Ingress load balances to available backend pod
  </Step>
  
  <Step title="Authentication middleware">
    JWT token validated via `authMiddleware.js`
  </Step>
  
  <Step title="Route handler processes request">
    Express route controller handles business logic
  </Step>
  
  <Step title="Database operations">
    Prisma Client queries PostgreSQL
  </Step>
  
  <Step title="Response returned">
    JSON response sent back to client
  </Step>
</Steps>

### WebSocket Message Flow

<Steps>
  <Step title="Client emits 'send_message' event">
    Socket.io client sends message data via WebSocket
  </Step>
  
  <Step title="Backend receives event">
    Socket handler in `handlers.js:114` processes the message
  </Step>
  
  <Step title="Message persisted to database">
    Prisma saves message to PostgreSQL with metadata
  </Step>
  
  <Step title="Event broadcast via Redis">
    Socket.io emits 'receive_message' to room
  </Step>
  
  <Step title="Redis distributes to all instances">
    All backend pods receive the event through pub/sub
  </Step>
  
  <Step title="All connected clients receive message">
    Every client in the channel/DM receives real-time update
  </Step>
</Steps>

## Socket.io with Redis Adapter

The architecture supports multiple backend instances through Socket.io's Redis adapter:

```javascript
// server.js:48
const io = new Server(server, {
    adapter: createAdapter(pubClient, subClient)
});
```

**How it works**:

1. Each backend instance maintains its own Socket.io server
2. Two Redis clients per instance (pub + sub)
3. When a message is broadcast to a room:
   - Socket.io publishes event to Redis
   - Redis broadcasts to all subscribed instances
   - Each instance delivers to its connected clients
4. Session affinity ensures stable WebSocket connections

<Warning>
Without Redis adapter, each backend instance would only know about its own connected clients, breaking multi-instance deployments.
</Warning>
